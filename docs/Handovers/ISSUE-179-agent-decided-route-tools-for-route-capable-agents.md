# ISSUE-179: Agent-decided route tools for route-capable agents

Status: Implemented

External: [Linear ZAR-149](https://linear.app/zara-voice/issue/ZAR-149/issue-179-agent-decided-route-tools-for-route-capable-agents)

## Work Completed

- Created Linear ZAR-149 and linked this local issue.
- Added local backlog and roadmap records for the follow-up architecture change.
- Decision: replace agent-attached route-policy classifier turns with an active-agent internal route action/tool. The active route-capable agent decides when enough context exists; runtime validates configured branch IDs and executes transfer semantics.
- Core contract slice: added `route_to_agent` action parsing, route menu projection on turn runtime packets, and provider-safe internal realtime route declarations through `zara_route_to_agent`.
- Sandwich runtime slice: removed agent-attached route-policy pre-classification from router traversal; route-capable agents now receive a safe `routeMenu`, enter action mode even without normal tools, and route only when the active model emits `route_to_agent`.
- Sandwich route execution validates configured branch IDs, ignores model-supplied targets, records packet intent/transfer/agent-selection facts, emits route announcement plus handoff events, and moves the next frontier to the routed agent.
- Premium realtime slice: session creation now includes normal agent tools plus an internal route tool for route-capable roles; provider route calls are handled separately from connector execution/grant validation, update active role/tool declarations after routing, and emit provider messages for the configured caller announcement.
- Premium realtime tool-loop filtering now passes only normal connector declarations to the agent tool executor, keeping internal route declarations out of integration grant checks.
- Builder/docs slice: added a tenant builder Router Agent preset that still creates the single underlying Agent node type with `routePolicy` enabled by default.
- Confirmed Router Agents remain normal tool-capable agents in the builder; the Tool action stays enabled after creating a Router Agent.
- Updated routing/tool/runtime/frontend/platform-admin docs to describe agent-attached routing as agent-decided via an internal route tool/action and runtime-validated, while standalone legacy intent routes remain classifier-backed until a future removal slice.
- Cleanup follow-up: removed retired seeded workspace fixture support and updated affected tests/docs so fresh tenants use only `workspace-default` unless a test explicitly models a user-created workspace.
- Toolbox follow-up: Router Agent is now selectable anywhere the normal Agent preset is selectable, including blank workflows. Router Agents created before specialist targets exist start with routing enabled and an empty branch list, with the inspector showing that another agent is needed before routing can be configured.
- Toolbox edge-case follow-up: when a branchless Router Agent is created before specialists, the inspector now seeds the first route branch once an available agent target exists, so the router does not get stuck in an unconfigurable state.
- Distinct-inspector follow-up: removed the Agent behavior selector. Normal Agents no longer expose route conversion controls, and Router Agents show routing fields directly in a Routing inspector section.
- Premium routing debug follow-up: diagnosed invoice-status calls that answered as the router instead of routing. The internal route tool was declared by the session path, but the premium realtime role prompt did not include route-specific behavior, so the model could behave like a normal support agent and ask for invoice details. Added manifest-derived Router Agent instructions to route before collecting branch-specific details, plus safe target role name/kind context so generic branch labels such as "Bill" still carry routing meaning.
- Role-derived routing follow-up: added a shared `@zara/core` route role profile resolver so built-in branch identity/copy comes from configured target role kind/profile, not agent display name. Builder-created branches now save deterministic labels such as `Billing`/`branch-billing` for billing targets even when the agent is named "Bill", and premium realtime prompts plus internal route tool descriptions project the same routing role label for legacy saved branch copy.
- Draft routing debug follow-up: diagnosed the latest invoice-status trace where OpenAI session diagnostics showed Jane with `sessionToolCount: 0`, `sessionToolsJson: []`, and no `# Routing` prompt section. Applying a specialist template to an existing Router Agent replaced the role object and silently dropped `routePolicy`, leaving only prose that said "route specialist work" with no machine-readable branches. Template apply/save now preserves existing Router Agent `routePolicy`, and normal agents still drop template-provided route policy metadata.
- Route-tool injection follow-up: fixed a core resolver gap where `buildRealtimeProviderToolDeclarations` only matched graph nodes with explicit `roleId`. Builder-created agents can use the node id as the role id, so a valid router with `roleId` omitted could compile a route policy but still get no `zara_route_to_agent` declaration. The resolver now matches `(node.roleId ?? node.id)` consistently with workflow publication.
- Published snapshot follow-up: preserved `routePolicy` metadata on published `VoiceAgentRole` snapshots so premium realtime's role-attached route-policy recovery path can reconstruct route policies when needed.
- Premium route-continuation follow-up: diagnosed the trace where Jane called `zara_route_to_agent`, Zara emitted "Handed off to Bill", but no Billing audio reached the caller. Two issues were present after the accepted route: route-time OpenAI `session.update` was still resending the target role's output voice/speed, which can be rejected after audio has already been produced; and the websocket bridge had already consumed the caller turn for Jane's spoken preamble, so Billing's follow-up audio was buffered instead of forwarded. Route-time updates now omit voice/speed, and validated handoff continuations reattach the last consumed caller turn before the follow-up provider response.
- Premium target-agent first-turn follow-up: diagnosed the next trace where the session successfully updated to Bill, but the only post-handoff response was "I'll connect you with Bill." The forced post-route `response.create` instructions were still framed around the route announcement, so the target model repeated the announcement instead of starting specialist work. The post-route response now tells the active target role that it is the routed specialist, includes the caller need summary, and instructs it to start helping in the same response instead of stopping after a transfer line.
- Premium bare-route announcement follow-up: diagnosed the next trace where Jane emitted a bare route tool call with no spoken preamble, Zara handed off to Bill, and Bill started helping but no caller-facing route message was audible. The post-route OpenAI continuation now carries the configured route announcement and tells the target role to speak it only if the immediately preceding assistant message did not already announce the handoff, then continue as the specialist in the same response.
- Premium deterministic announcement follow-up: diagnosed the latest trace where the caller confirmed "Yes", Jane emitted only the internal route tool call, and Bill continued without any audible route acknowledgement. The soft conditional wording let the target model treat "thanks for contacting the billing team" as enough. OpenAI route handling now inspects the raw `response.done` output: if a message item appears before the matching route tool call, the continuation skips repeating the announcement; otherwise the continuation must begin with the exact configured handoff sentence before Bill starts specialist work.
- Premium source-announced handoff correction: accepted the product correction that Jane, not Bill, must always speak the handoff before routing. Bare OpenAI route tool calls now delay the role switch: Zara returns the route tool output, asks the current source role to say the exact configured announcement and stop, stores the validated route as pending, and only applies the handoff/session update plus Bill's first response after Jane's announcement response completes.

## Implementation Plan

1. Core contract slice: add route action parsing, route menu projection into agent turn context, and internal realtime route declarations without exposing graph targets or credentials.
2. Sandwich runtime slice: stop classifying agent-attached route policies in the router; enable agent-action mode for route-capable agents even when no normal tools exist; validate route actions and switch active agent through the existing packet/transfer event path.
3. Premium realtime slice: include internal route declarations for route-capable roles, handle provider route calls separately from connector tools, update the active session role/tools after validated route, and emit the configured caller announcement before the target agent continues.
4. Builder/docs slice: present Router Agent as an intuitive agent preset while preserving normal tools, and update routing docs to describe agent-decided runtime-validated routing.
5. Verification slice: run focused core/runtime/premium tests and typecheck; update issue status and handover with final commands and risks.

## Risks

- The working tree already has many unrelated modified files from previous slices; ISSUE-179 changes must stay surgical and avoid reverting others' work.
- Premium realtime must keep provider credentials, provider URLs, connector secrets, and graph target IDs out of browser/provider-visible surfaces.
- Internal route tools must not be treated as integration grants or connector tools.
- Route-capable agents with normal tools need both normal tool declarations and the internal route declaration.
- Premium realtime route behavior still depends on provider tool-choice behavior; the prompt now explicitly tells the route-capable role to route first when a configured branch clearly matches.
- Duplicate same-role specialists can still be ambiguous if future UI allows more than one active branch for the same built-in role; branch IDs currently derive from role intent keys and should get a deterministic specialization suffix before multi-branch same-role routing is added.
- Full focused websocket suite passed after the workspace fixture cleanup.
- Diagnostics now distinguish "model did not choose to route" from "runtime did not provide routing." A session with no `# Routing` section and `sessionToolCount: 0` means route policy was absent or failed active-role matching before provider setup.
- Premium route continuations can produce two provider responses for one caller turn: the router's spoken preamble and the routed agent's continuation. The bridge must preserve or restore caller-turn confirmation across that continuation so target-agent audio is not buffered as orphan output.
- A successful premium route must produce both caller awareness and the target agent's first working turn in the right order. Jane/the source Router Agent owns the caller-facing handoff. Bare OpenAI route tool calls must pause routing, have the source role speak the exact configured announcement, and only then apply the target role switch.

## Decisions

- Router Agent is a distinct builder preset and inspector experience, not a distinct runtime node type.
- Route-capable agents can keep normal tools. Routing is an additional capability, not a replacement for tool use.
- The active model owns the "enough context" decision by choosing or not choosing the route action/tool.
- Runtime remains authoritative for branch validation, target resolution, loop/language guards, announcements, packet facts, and active-role switching.
- The Router Agent preset derives its initial target menu from existing agent nodes and excludes the selected source agent so a router created after Front Desk does not default its first branch back to Front Desk.
- Router Agents may be created before specialist targets exist; their initial route policy remains branchless until another agent is available, then the inspector seeds the first configurable branch from that target.
- Premium realtime Router Agent prompts must include the same manifest-derived route menu as the internal route tool, safe target role name/kind context, and must instruct the model not to collect branch-specific details before routing.
- Built-in Router Agent branch identity/copy is role-derived. Agent display names such as "Bill" remain persona/target labels, while branch labels, intent keys, descriptions, examples, and prompt routing-role labels come from the configured target role kind/profile. Custom roles fall back to the configured role name until explicit custom route metadata exists.
- Specialist templates must not change whether the selected node is a normal Agent or Router Agent. Applying/saving templates updates the role persona/config while preserving an existing Router Agent `routePolicy`.
- OpenAI realtime voice/speed are initial-session settings for this implementation. Route-time role switches update prompt/language/tools only; they do not attempt to switch provider output voice mid-call.
- Post-route OpenAI `response.create` instructions should identify the active target role, include the caller need summary, avoid repeating the source handoff, and start specialist work. If the route tool call was bare, Zara must first issue a source-agent announcement response and defer the target role switch until that source announcement completes.
