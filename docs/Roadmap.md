## Integrations

OAuth framework, encrypted credential storage, Zendesk, HubSpot, Google Workspace, Notion, webhook tools, health/revocation, and permission grants.

Knowledge-source v1 decision: support both snapshot imports and recurring sync sources. Snapshot imports cover manual text, PDFs, single URLs, and one-time provider imports. Recurring sync sources stay connected to the provider, refresh manually or on a daily cadence, create review-gated update drafts, and never change active runtime knowledge until an operator approves the new snapshot and republishes or updates the allowed workflow scope. Hourly and custom schedules are intentionally out of v1. Confirmed source deletions create deletion/stale review drafts while the current approved snapshot remains active until an operator approves deletion or disables the record; auth or permission failures degrade the source sync instead of deleting active knowledge.

Knowledge record taxonomy note: expand tenant knowledge beyond the current policy/FAQ baseline to include FAQ, policy, procedure, troubleshooting, pricing, escalation, legal/compliance, and general reference records. Conflict handling should use source priority plus conflict warnings, and block publish only for unresolved high-risk conflicts such as authoritative policy, pricing, legal/compliance, billing/payment, safety, or escalation records.

Knowledge classification note: imported knowledge should use hybrid classification. Manual entries choose a record type directly, defaulting to general reference. Imported articles, pages, files, and URLs may receive suggested record types, but high-risk suggestions such as pricing, legal/compliance, policy, and escalation require explicit operator confirmation before activation; runtime retrieval never includes unconfirmed draft knowledge.

Knowledge review unit note: review happens at the extracted-record level, not at raw embedding chunk level. Model the hierarchy as source, fetched/imported snapshot, extracted knowledge records, then internal chunks/embeddings. Tenant UI should show source status, snapshot diffs, record-level approval cards, source excerpts, scope, confidence, and conflict markers while keeping embedding chunks internal.

Knowledge runtime scope note: published workflow manifests should freeze the allowed knowledge scope definition, including workspace, source IDs, record types, and workflow/role permissions, while allowing newly approved records inside that scope to become available to new calls without republishing. Active calls keep the retrieval snapshot they started with. Adding a new source or broader scope requires republish or an explicit audited route update.

Knowledge runtime retrieval note: v1 calls should answer from indexed, approved Zara knowledge only, not live provider knowledge search. Operational tools may still perform explicit provider lookups such as order, subscription, ticket, or calendar status, but unreviewed live provider search results must not enter the agent's knowledge lane during a call.

Knowledge sensitivity note: ingestion should scan extracted records for sensitive content and label records such as PII, credentials, payment, health, legal, or internal-only. Obvious credentials, secrets, API keys, and passwords must not be activated into runtime knowledge. High-risk sensitive records require explicit admin approval, source excerpts should avoid unnecessary full sensitive text rendering, and sensitive knowledge approvals must be audited.

Knowledge and integration approval authority note: use the existing tenant/workspace roles assigned in Settings rather than introducing a separate approval-role system. Tenant owners/admins can approve organization-wide risky actions; workspace admins can approve within their workspace; builders may request grants but cannot approve high-risk writes or sensitive/authoritative knowledge; operators may approve low-risk memory or knowledge drafts only when workspace policy allows; viewers cannot approve. Risky approvals include write tools, sensitive knowledge, authoritative pricing/policy/legal/escalation records, source deletion approvals, organization-wide connection promotion, and post-call sync to external systems. Approval records must include actor, role, workspace, reason, before/after state, and timestamp.

Integration grant UX note: a provider connection is not a blanket permission grant. Agent tools, knowledge-source ingestion, and post-call sync each need explicit scoped grants, but tenant UI should present these as clear capability toggles and guided setup steps instead of raw permission records. Default recommendations should be safe and understandable, such as read knowledge for selected workspace, allow selected workflow tools, and require approval for risky write actions.

Integration OAuth scope note: capabilities must validate against granted provider scopes before grants, publish, and runtime execution. If a connection lacks required scopes, tenant UI should disable that capability with a clear reconnect action requesting only the missing scopes, preserve the existing connection until reconnect succeeds, and block publish for tools or sync jobs bound to insufficiently scoped credentials.

Integration connection scope note: support both tenant-wide and workspace-owned provider connections to reduce tenant setup friction. Tenant-wide connections are available for scoped grants across workspaces, while workspace-owned connections stay visible and usable only inside their workspace by default. UI should present this as a simple setup choice such as "Use across organization" or "Use only in this workspace"; runtime and grant checks still enforce tenant, workspace, workflow, role, capability, and scope boundaries. Tenant owners/admins may promote a workspace-owned connection to organization-wide after review, but promotion changes availability only, creates no automatic new capability grants, and must be audited.

Integration revoke/delete note: revoke disables credential use immediately while preserving history, grants, imported knowledge snapshots, sync records, and audit trail. Delete is allowed only after active grants, sync jobs, routes, or dependent workflow/tool bindings are removed or reassigned. Revoked connections pause recurring knowledge sync, leave approved imported snapshots visible as source-disconnected, invalidate bound tool nodes for publish, cause runtime tools to fail safely with auth-revoked classification, and pause post-call sync visibly until reconnect or reassignment.

Integration setup preset note: tenant setup should offer previewable presets such as support agent, sales agent, and ecommerce support. Presets may preselect safe capability grants, source scopes, and approval posture, but operators must be able to review and edit the generated grants before saving. Risky write tools should default to approval-required. Preset generation must stay catalog-driven: provider existence alone is not enough to expose capability lanes such as post-call sync.

Integration setup copy note: workspaces may copy integration setup templates, including enabled capabilities, selected tools, approval posture, knowledge source types, sync cadence, post-call sync settings, and recommended scopes. Copying must require target workspace, connection, source/category, grant, and risky-write confirmation. Credentials, OAuth grants, and workspace-owned source access must not be silently cloned across workspaces.

Provider registry architecture note: ISSUE-156 implements the hybrid capability registry foundation. Shared/public metadata lives in `@zara/core` for provider IDs, labels, categories, capabilities, safe setup schema, tool IDs/names, risk posture, logo tokens, knowledge-source flags, documentation references, and docs-verified dates. API-owned registry metadata retains provider base URLs, auth header construction, secret schema IDs, and executor IDs server-side. ISSUE-157 migrates the tenant integrations page and workflow tool selection to consume `GET /organizations/:orgId/integrations/catalog` instead of maintaining separate hardcoded built-in provider/tool lists.

Provider tool authoring note: built-in provider tools should be curated Zara business actions, not raw provider API operations exposed to workflow builders. Generated clients or provider OpenAPI types may be used internally, but tenant-facing tools need stable Zara input/output schemas, risk posture, approval defaults, required scopes, rate-limit behavior, and tests. API URLs, paths, auth headers, and provider-specific payload shapes remain hidden for built-in providers; custom webhook tools are the explicit user-configurable exception.

Provider contract testing note: every built-in provider tool needs deterministic mocked contract tests that assert the documented provider method, path, query/body shape, auth headers, input validation, normalized output schema, provider error mapping, rate-limit behavior, approval posture, tenant/workspace isolation, and secret redaction. Registry metadata should carry provider documentation references and a docs-verified date. Optional live provider smoke tests may run only when test credentials are configured and should not block ordinary CI on third-party availability.

Integration runtime failure note: provider tool failures during calls should return structured outcomes instead of letting the model improvise. Classify failures as auth revoked, permission denied, not found, rate limited, provider unavailable, timeout, or validation error; expose them in live monitor, call summaries, integration health, and trace events. Runtime fallback should be safe and explicit, with human escalation preferred for billing, payment, order, or other high-risk flows.

Integration side-effect safety note: write tools must use a Zara side-effect ledger plus provider idempotency keys when supported. Store pending, succeeded, failed, and unknown outcomes; treat post-send timeouts as unknown rather than retryable failure; block blind automatic retries for unknown writes; and have post-call sync consult the ledger before creating tasks, tickets, notes, events, or summary posts. Operators may manually retry uncertain writes from an audited UI.

Integration implementation sequence note: implement the provider registry foundation before adding the popular expansion batch. First migrate Zendesk, HubSpot, Google Workspace, Notion, and webhook tools into the hybrid registry and API-served catalog; then add capability grants and org/workspace connection scope; then add the knowledge-base add/import UX with snapshot plus daily sync and review-gated activation; then add Salesforce, Slack, Microsoft 365, Intercom, Shopify, and Stripe. Do not add new providers on top of the current hardcoded frontend/backend catalogs.

Integration expansion batch note: after the provider registry lands, the first popular-integration expansion batch should target Salesforce, Slack, Microsoft 365, Intercom, Shopify, and Stripe. Salesforce should start with account/contact/case lookup plus approval-required additive writes only, such as create task, create case, and add call note; pipeline mutation, owner changes, destructive updates, and deletes are out of v1. Slack should start with bounded escalation, failed-call/provider-health alerts, and configurable post-call summary posts to configured destinations only; arbitrary agent-generated messages, arbitrary DMs, and channel-history reads are out of v1. Microsoft 365 v1 is Outlook calendar only, with availability reads and event creation; email send/read, mailbox search, and broad Graph scopes are out of v1. Intercom should start with user/company/open-conversation lookup, internal note or call-summary creation, and Articles ingestion; external replies, conversation closing, assignment changes, and user/company field mutation are out of v1. Shopify v1 is read-only customer, order, fulfillment, and shipping-status lookup; refunds, cancellations, address edits, draft order creation, discount changes, and inventory changes are out of v1. Stripe should be read-only customer/subscription, invoice, and payment-status lookup only. Stripe write actions such as refunds, cancellations, payment-method changes, invoice creation, coupon changes, or payment retries are out of v1.

Future registry-stability note: once the capability-based provider registry is stable, add follow-up knowledge-source connectors for full website crawling, SharePoint, Confluence, Freshdesk Solutions, and Salesforce Knowledge. Intercom Articles has its v1 baseline in ISSUE-165; any future Intercom work should be deeper ingestion coverage rather than a new source-picker placeholder.
